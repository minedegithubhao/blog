# 乘风博客功能介绍与开发规范

最后更新：2026-04-28

本文档是乘风博客前期讨论形成的完整决策文档。后续人类开发者和 Agent 开发机器人必须优先阅读本文档，再进行设计、编码、重构、接口扩展或部署调整。

本文档目标：

- 明确乘风博客 V1 做什么、不做什么。
- 统一前端、后端、数据库、Agent、部署、接口和编码规范。
- 把规则写到 AI 可以直接执行，减少二次解释成本。
- 避免后续 Agent 自行发明架构、接口、命名和目录结构。

# 一、项目概述

## 1.1 项目定位

乘风博客 V1 是一个轻量级内容分发与 Agent 体验平台。

它不是大而全的 CMS，也不是纯展示型博客。它的核心价值是：

- 用博客文章承接内容流量。
- 用 Agent 专栏展示可体验的 AI 能力。
- 用用户登录、角色、额度、调用记录形成最小闭环。
- 用后台管理支撑文章、分类、文件、Agent、用户、额度和站点配置。

## 1.2 核心链路

1. 管理员登录后台。
2. 管理员创建分类、上传文件、发布文章。
3. 普通用户访问前台首页、文章列表和文章详情。
4. 普通用户进入 Agent 专栏，查看 Agent 列表和详情。
5. 用户注册或登录后获得默认额度。
6. 用户体验 Agent，系统校验权限、校验额度、调用 Python Agent Runtime。
7. Agent 响应通过 SSE 流式返回前端。
8. 系统记录调用记录、消耗额度、保留会话和消息。
9. 管理员可在后台调整用户额度并查询调用记录。

## 1.3 当前约束

- 当前阶段优先单机部署和本地联调。
- 后台用户规模预计 3 到 5 人。
- 前台同时在线预计 20 到 50 人。
- 峰值 QPS 预计 3 到 5。
- 性能瓶颈主要来自 LLM 或 Agent 长连接，不来自普通 CRUD。
- 一期必须跑通主链路，不追求复杂社区化、商业化和多租户能力。

# 三、技术栈

## 3.1 前端技术栈

blog_web 当前主技术栈：

- 框架：Next.js 15 + React 19。
- 路由：App Router。
- 语言：TypeScript。
- 样式：Tailwind CSS 4 + PostCSS。
- 组件：shadcn/ui + Radix UI。
- 图标：lucide-react。
- 表单：react-hook-form。
- 校验：zod + @hookform/resolvers。
- 图表和交互补充：recharts、sonner、embla-carousel-react、date-fns。
- 包管理：pnpm。

前端技术约束：

- 不再同时维护多套组件体系。
- 后台和前台统一使用 shadcn/ui + Radix UI + Tailwind CSS + lucide-react。
- 不再引入 Ant Design、Element、MUI 等额外大型 UI 框架。
- 所有可复用交互组件必须优先使用前端 UI 框架原生组件，统一从 `src/components/ui` 消费。
- 业务页面禁止为了局部样式差异重复手写按钮、输入框、下拉、弹窗、分页、表格、Tabs、Tooltip、Badge、Toast 等已有官方组件。
- 如果 shadcn/ui 当前缺少某个组件，必须先按官方方式补齐到 `src/components/ui`，再由业务页面复用；禁止在业务目录中直接散落封装一套 look-alike 组件。
- 新页面优先复用现有 `components/ui`。
- 如果必须新增依赖，先向用户说明理由并等待确认。

## 3.2 后端技术栈

后端主技术栈：

- Java：JDK 17。
- 框架：Spring Boot 3.x。
- ORM：MyBatis-Plus。
- 数据库：MySQL 8.x。
- 缓存：Redis 7.x。
- Maven: 3.9.6
- 接口风格：RESTful + SSE。
- 容器化：Docker + Docker Compose。
- 监控：Spring Boot Actuator + 日志，一期不强制 Prometheus。

后端技术约束：

- 一期保持 Spring Boot 单体应用。
- 不引入 WebFlux。
- 不引入微服务框架。
- 不引入复杂工作流引擎。
- 外部 Agent 调用必须有超时、线程池隔离和调用记录。

## 3.3 Agent Runtime 技术栈

Agent Runtime 用 Python 实现，独立于 Spring Boot。

职责：

- 承载具体 Agent 执行逻辑。
- 对接 LLM、工具、知识库或外部 API。
- 向 Spring Boot 返回普通响应或流式响应。

边界：

- Python Runtime 不负责登录。
- Python Runtime 不负责角色判断。
- Python Runtime 不负责用户额度。
- Python Runtime 不直接暴露给前端。
- Python Runtime 不直接操作博客业务数据库。

# 四、部署架构

## 4.1 当前阶段部署组件

当前阶段采用 Docker Compose 单机部署。

组件清单：

- `blog-web`：Next.js 前端，负责前台和后台页面。
- `blog-server`：Spring Boot 后端，负责业务 API、认证、权限、额度、文章、Agent 网关。
- `python-agent-runtime`：Python Agent 执行服务，负责具体 Agent 调用。
- `mysql`：MySQL 8.x，负责业务数据持久化。
- `redis`：Redis 7.x LTS/稳定版，负责缓存、限流、会话辅助和临时状态。
- `nginx`：可选组件，用于统一入口、静态资源代理和反向代理。

## 4.2 请求流转

普通博客请求：

```text
Browser
  -> blog-web
  -> blog-server
  -> mysql / redis
  -> blog-server
  -> blog-web
  -> Browser
```

Agent 非流式请求：

```text
Browser
  -> blog-web
  -> blog-server
  -> 校验登录
  -> 校验角色
  -> 校验额度
  -> python-agent-runtime
  -> 记录调用结果
  -> 扣减额度
  -> Browser
```

Agent SSE 流式请求：

```text
Browser
  -> blog-web
  -> blog-server SseEmitter
  -> agent-call-pool
  -> python-agent-runtime stream
  -> blog-server 转发 token/event
  -> Browser
```

## 4.3 组件职责

`blog-web`：

- 只负责页面渲染和用户交互。
- 只调用 `blog-server`。
- 不直接调用 MySQL、Redis、Python Runtime。
- 不在前端保存密钥。

`blog-server`：

- 是业务系统唯一后端入口。
- 负责认证、权限、额度、文章、分类、文件、Agent 元数据。
- 负责调用 Python Runtime。
- 负责调用记录、异常转换、SSE 转发。

`python-agent-runtime`：

- 只负责执行 Agent。
- 必须提供健康检查接口。
- 必须支持请求超时和取消。
- 返回结构必须稳定，不能把第三方原始异常直接暴露给前端。

`mysql`：

- 保存强一致业务数据。
- 保存用户、文章、分类、Agent、额度、调用记录。

`redis`：

- 保存短期缓存。
- 保存限流计数。
- 保存临时会话状态。
- 不作为最终业务数据来源。

## 4.4 运维预期

部署方式：

- Docker Compose 一键启动。
- 一期单机部署。
- 后续可拆分服务，但当前不提前拆。

JVM：

- Spring Boot 容器必须设置 JVM 内存上限。
- 默认建议：`-Xmx512m`。
- 目的是防止容器被 OOM Kill。

用户量：

- 20 到 50 人同时在线。
- 单机部署够用。

QPS：

- 峰值 3 到 5 QPS。
- 瓶颈在 LLM 长连接，不在普通 CRUD。

监控：

- 一期使用 Spring Boot Actuator + 应用日志。
- 后期再接 Prometheus + Grafana。

扩容：

- 一期单机。
- 架构上不堵死扩容路径。
- 未来拆分时优先拆 Agent Runtime、文件存储、读多写少查询。

# 五、功能清单

## 5.1 前台页面

前台页面必须包含：

- `/`：首页。
- `/blog`：博客首页或博客频道页。
- `/blog/posts`：文章列表。
- `/blog/posts/[id]`：文章详情。
- `/blog/agent`：Agent 专栏列表。
- `/blog/agent/[id]`：Agent 详情和体验入口。
- `/login`：前台登录。
- `/register`：前台注册。
- `/me`：简化个人页。

## 5.2 后台页面

后台页面必须包含：

- `/dashboard`：后台首页。
- `/dashboard/articles`：文章管理。
- `/dashboard/categories`：分类管理。
- `/dashboard/tags`：标签管理。
- `/dashboard/users`：用户管理。
- `/dashboard/roles`：角色管理。
- `/dashboard/menus`：菜单管理。
- `/dashboard/dict`：字典管理。
- `/dashboard/agents`：Agent 管理。
- `/dashboard/quotas`：额度管理。
- `/dashboard/agent-call-records`：Agent 调用记录。
- `/dashboard/files`：文件管理。
- `/dashboard/site-config`：站点配置。

## 5.3 数据对象

核心数据对象：

- 后台账号：用于登录后台和维护内容。
- 前台用户：用于登录前台、体验 Agent、绑定额度。
- 角色：控制用户身份、Agent 可用范围和默认额度策略。
- 菜单：后台菜单和权限入口。
- 字典：系统枚举型配置。
- 文章：博客内容主体。
- 分类：文章基础组织方式，V1 只做单层分类。
- 标签：文章辅助组织方式。
- 文件资源：文章封面、正文图片、Agent 封面。
- Agent：AI 能力展示和体验对象。
- Agent 会话：用户与某个 Agent 的对话上下文。
- Agent 消息：会话内消息。
- 角色 Agent 策略：角色是否能使用某 Agent，以及默认额度。
- 用户 Agent 额度：用户对某 Agent 的额度覆盖配置。
- Agent 调用记录：每次 Agent 调用的审计和计量记录。
- 站点配置：站点名称、Logo、首页文案、首页头像、社交链接、首页统计展示值、页脚等。

V1 账号实现补充说明：

- V1 前台用户与后台账号临时共用 `blog_account` 一张表。
- 后续如有必要，再拆分为前台用户表和后台账号表。

建模原则：

- Entity 对应数据库表。
- DTO 接收请求。
- VO 返回前端。
- 不允许 Controller 返回 Entity。
- 不允许 Mapper 接收 DTO。
- 不允许前端依赖数据库字段名。

# 六、后端代码组织规范

## 6.1 总包结构

后端统一根包：

```text
com.canbe.blog
├── BlogApplication.java
├── common
├── config
├── security
├── infrastructure
├── user
├── content
├── agent
├── quota
├── file
└── system
```

模块职责：

- `common`：通用结果、异常、常量、工具、校验。
- `config`：Spring 配置、MyBatis-Plus 配置、线程池配置、跨域配置。
- `security`：认证、Token、当前用户上下文、登录拦截。
- `infrastructure`：外部系统适配器，例如 Agent Runtime、Redis、对象存储、LLM。
- `user`：前台用户、后台账号、角色。
- `content`：文章、分类、标签。
- `agent`：Agent 元数据、会话、消息、调用入口。
- `quota`：角色策略、用户额度、额度扣减、额度查询。
- `file`：文件上传、文件记录。
- `system`：菜单、字典、站点配置、后台看板。

## 6.2 模块内部结构

每个业务模块必须按以下结构组织：

```text
module
├── controller
├── service
│   └── impl
├── mapper
├── entity
├── dto
├── vo
├── convert
└── enums
```

命名规则：

- Controller：`AdminXxxController`、`PublicXxxController`。
- Service：`XxxService`。
- ServiceImpl：`XxxServiceImpl`。
- Mapper：`XxxMapper`。
- Entity：`Xxx`。
- DTO：`XxxCreateDTO`、`XxxUpdateDTO`、`XxxQueryDTO`、`XxxCommandDTO`。
- VO：`XxxVO`、`XxxDetailVO`、`XxxPageVO`、`XxxOptionVO`。
- Convert：`XxxConvert`。
- Enum：`XxxStatusEnum`、`XxxTypeEnum`。

## 6.3 分层职责

Controller 只允许：

- 接收 HTTP 请求。
- 使用 `@Validated` 校验 DTO。
- 获取当前用户上下文。
- 调用本模块 Service。
- 返回统一 `Result<T>`。

Controller 禁止：

- 禁止注入 Mapper。
- 禁止写业务规则。
- 禁止写额度扣减。
- 禁止拼 SQL。
- 禁止直接调用其他模块 Mapper。
- 禁止直接调用外部系统 Client。
- 禁止返回 Entity。

Service 允许：

- 编排业务流程。
- 判断业务规则。
- 控制事务。
- 调用本模块 Mapper。
- 调用其他模块 Service。
- 调用 `infrastructure` 适配器。
- 完成 DTO、Entity、VO 转换。

Service 禁止：

- 禁止接收 `HttpServletRequest` 作为业务参数。
- 禁止返回 MyBatis-Plus 原始 `Page` 给 Controller。
- 禁止把 Entity 直接返回给 Controller。
- 禁止跨模块调用其他模块 Mapper。
- 禁止直接解析 JWT。

Mapper 只负责数据库访问。

Mapper 允许：

- 继承 `BaseMapper<Entity>`。
- 编写本表查询。
- 编写本模块聚合查询。
- 使用 XML 写复杂 SQL。

Mapper 禁止：

- 禁止写业务判断。
- 禁止调用其他 Mapper。
- 禁止调用 Service。
- 禁止跨模块操作其他模块表。
- 禁止绕过 Service 直接返回给 Controller。

Convert 只做对象转换。

Convert 禁止：

- 禁止注入 Service。
- 禁止注入 Mapper。
- 禁止查数据库。
- 禁止写业务判断。

## 6.4 跨模块调用规则

允许调用链：

```text
controller -> same module service
service -> same module mapper
service -> other module service
service -> infrastructure client
```

禁止调用链：

```text
controller -> mapper
controller -> other module service
service -> other module mapper
mapper -> mapper
mapper -> service
entity -> service
convert -> service
```

合法示例：

```text
agent.service.AgentChatService
  -> quota.service.QuotaCheckService
  -> quota.service.QuotaDeductService
  -> infrastructure.agent.AgentRuntimeClient
```

非法示例：

```text
agent.service.AgentChatService
  -> quota.mapper.UserAgentQuotaMapper
```

## 6.5 事务规则

- 写操作 Service 方法必须加 `@Transactional`。
- 只读查询方法加 `@Transactional(readOnly = true)` 或不加事务。
- 一个业务流程只在最外层 Service 方法加事务。
- Controller 禁止加 `@Transactional`。
- 捕获异常后仍需回滚时，必须重新抛出异常或显式设置回滚。

# 七、后端编码规范

本节基于《阿里巴巴 Java 开发手册》提炼，结合乘风博客后端实际约束压缩成 AI 可直接执行的规则。

## 7.1 命名规范

1. Java 命名禁止以下划线或美元符号开头、结尾。
2. 禁止中文命名、拼音命名、拼音英文混用。
3. 类名使用 `UpperCamelCase`。
4. 方法名、参数名、成员变量、局部变量使用 `lowerCamelCase`。
5. 常量使用 `UPPER_UNDERSCORE_CASE`。
6. 抽象类以 `Abstract` 或 `Base` 开头。
7. 异常类以 `Exception` 结尾。
8. 测试类以被测类名开头，以 `Test` 结尾。
9. 枚举类以 `Enum` 结尾，枚举项全大写下划线命名。
10. Service 方法动作必须明确：单个查询用 `get`，列表查询用 `list`，统计用 `count`，新增用 `create` 或 `save`，修改用 `update`，删除用 `delete` 或 `remove`。

## 7.2 异常处理规范

1. 能通过前置判断规避的异常，必须先判断再执行。
2. 禁止用 `try-catch` 处理空指针、数组越界、集合越界等可预防问题。
3. 异常不能作为正常业务流程控制。
4. `catch` 后必须处理，禁止空 `catch`。
5. `catch` 处理方式只能是转换为业务异常、记录日志后返回明确错误、继续向上抛出。
6. 禁止只打印 `e.getMessage()`。
7. 禁止直接抛出裸 `RuntimeException`、`Exception`、`Throwable`。
8. 业务异常必须使用有语义的自定义异常。
9. 对外 HTTP 接口统一返回 `Result<T>` 和错误码。
10. 资源、流、连接必须使用 `try-with-resources` 或在 `finally` 中关闭。
11. `finally` 中禁止 `return`。

## 7.3 日志规范

1. 日志统一使用 `slf4j` 门面。
2. 禁止业务代码直接依赖 Logback、Log4j 等具体实现。
3. 日志必须包含可定位现场的信息。
4. Agent 相关日志必须优先包含 `requestId`、`userId`、`agentId`、`sessionId`。
5. 文章相关日志必须优先包含 `articleId`、`categoryId`、`operatorId`。
6. 记录异常时必须把异常对象作为最后一个参数传入。
7. 日志输出必须使用占位符，禁止字符串拼接。
8. 生产环境禁止输出 `debug` 日志。
9. `info` 只记录关键业务节点。
10. `warn` 记录可恢复异常或用户输入问题。
11. `error` 只记录系统异常和不可恢复失败。
12. 禁止在高频循环中打印大量日志。

## 7.4 并发规范

1. 禁止手动 `new Thread`。
2. 禁止使用 `Executors` 快捷工厂创建线程池。
3. 必须使用显式 `ThreadPoolExecutor`。
4. 线程池必须配置核心线程数、最大线程数、队列长度、线程名、拒绝策略。
5. 线程池必须按业务隔离，例如 `agent-call-pool`、`file-upload-pool`。
6. 外部 Agent 调用不得复用 Web 容器线程池长期阻塞。
7. 多资源加锁必须固定顺序。
8. 并发更新同一业务记录必须使用乐观锁版本号、数据库条件更新或分布式锁。
9. 禁止无保护地先查后改。
10. 日期时间使用 `java.time`，不要使用线程不安全的静态 `SimpleDateFormat`。
11. 计数优先使用 `AtomicLong` 或 `LongAdder`。
12. 使用 `ThreadLocal` 后必须在 `finally` 中清理。

## 7.5 MySQL 编码规范

1. 表名和字段名只使用小写字母、数字、下划线。
2. 禁止大写表名、复数表名、MySQL 保留字。
3. 布尔数据库字段使用 `is_xxx`。
4. Java POJO 布尔属性不要加 `is` 前缀。
5. 每张业务表必须包含 `id`、`gmt_create`、`gmt_modified`。
6. 金额和精确小数必须使用 `decimal`。
7. 禁止使用 `float`、`double` 存储金额或精确小数。
8. 查询禁止 `select *`。
9. 禁止左模糊和全模糊搜索。
10. 禁止超过三张表 Join。
11. SQL 参数必须使用 `#{}` 绑定。
12. 禁止 `${}` 拼接用户入参。
13. 删除和批量更新前必须明确条件。
14. 禁止无条件更新、无条件删除。
15. 业务代码禁止使用 `TRUNCATE`。

# 八、数据库性能规范

## 8.1 通用建表规则

每张业务表必须包含：

```sql
id bigint unsigned not null primary key,
gmt_create datetime not null,
gmt_modified datetime not null,
is_deleted tinyint unsigned not null default 0
```

通用字段规则：

- 主键统一使用 `id`。
- 创建时间使用 `gmt_create`。
- 修改时间使用 `gmt_modified`。
- 逻辑删除使用 `is_deleted`，0 表示未删除，1 表示已删除。
- 状态字段使用 `status`。
- 类型字段使用 `type`。
- 排序字段使用 `sort_order`。
- 乐观锁字段使用 `version`。

## 8.2 索引设计原则

- 业务唯一字段必须建唯一索引。
- 唯一索引命名：`uk_字段名`。
- 普通索引命名：`idx_字段名`。
- 组合索引优先放等值条件字段。
- 高频查询必须按实际查询条件设计索引。
- `order by` 字段尽量放在组合索引最后。
- 低区分度字段不单独建索引。
- 索引不是越多越好，新增索引必须说明查询场景。

## 8.3 核心表索引建议

文章表：

- `idx_status_gmt_create`：前台文章列表。
- `idx_category_status_gmt_create`：分类文章列表。
- `idx_slug` 或 `uk_slug`：固定链接。

分类表：

- `uk_slug`：分类路径唯一。
- `idx_status_sort_order`：前台分类展示。

标签表：

- `uk_name`：标签名唯一。

用户表：

- `uk_username`：用户名唯一。
- `uk_email`：邮箱唯一，未启用邮箱时可暂缓。

Agent 表：

- `idx_status_sort_order`：前台 Agent 列表。
- `uk_code`：Agent 编码唯一。

额度表：

- `uk_user_agent`：同一用户同一 Agent 只有一条额度记录。

调用记录表：

- `idx_user_gmt_create`：按用户查调用记录。
- `idx_agent_gmt_create`：按 Agent 查调用记录。
- `idx_request_id`：按请求追踪调用。

## 8.4 大表预判

一期可能变大的表：

- `agent_call_record`。
- `agent_message`。
- `article_view_log`，如果后续增加访问日志。

处理策略：

- 一期不分库分表。
- 调用记录和消息表必须保留时间索引。
- 后台查询必须带时间范围。
- 分页最大 `pageSize` 为 100。
- 深分页必须改成游标分页或延迟关联。

## 8.5 分页规则

- `page` 从 1 开始。
- `pageSize` 默认 20。
- `pageSize` 最大 100。
- `count = 0` 时直接返回空列表。
- 禁止无条件导出全表。
- 后台列表必须提供必要筛选条件。

# 九、Agent 架构规范

## 9.1 Agent 专栏与 Spring Boot 的关系

Spring Boot 是 Agent 专栏的业务网关，不是 Agent 执行引擎。

Spring Boot 负责：

- Agent 元数据管理。
- 前台 Agent 列表和详情。
- 用户登录校验。
- 角色权限校验。
- 额度校验。
- 调用记录。
- 会话和消息持久化。
- SSE 流式转发。
- 异常转换。

Python Agent Runtime 负责：

- 执行具体 Agent。
- 调用 LLM。
- 调用工具。
- 访问知识库。
- 返回流式或非流式结果。

前端只允许调用 Spring Boot。

禁止：

- 前端直接调用 Python Runtime。
- Python Runtime 直接判断博客用户权限。
- Python Runtime 直接扣减额度。
- Python Runtime 直接写博客业务表。

## 9.2 SSE 与 WebFlux 决策

当前阶段使用 Spring MVC + `SseEmitter`。

不引入 WebFlux 的理由：

- 当前系统是低 QPS 单体应用。
- 主要阻塞来自外部 Agent 长连接。
- 使用独立线程池隔离即可满足一期需求。
- 引入 WebFlux 会增加技术栈复杂度和改造成本。

SSE 规则：

- Controller 创建 `SseEmitter` 后立即返回。
- Agent 调用提交到 `agent-call-pool`。
- 后台线程读取 Python Runtime 流并写入 `SseEmitter`。
- 完成时调用 `emitter.complete()`。
- 异常时发送错误事件并调用 `completeWithError()`。
- 超时和客户端断开时必须取消下游调用。

## 9.3 线程池隔离

必须配置独立线程池：

```text
agent-call-pool
coreSize: 8
maxSize: 16
queueCapacity: 100
threadNamePrefix: agent-call-
rejectedExecutionHandler: AbortPolicy 或自定义返回 AGENT_BUSY
```

隔离规则：

- Agent 调用必须进入 `agent-call-pool`。
- 文件处理可使用 `file-task-pool`。
- 定时任务可使用 `scheduled-task-pool`。
- 禁止所有异步任务共用一个默认线程池。
- 外部服务卡住时，只能占满对应业务线程池，不能拖垮 Web 请求线程。

## 9.4 超时、重试、容错

超时规则：

- 连接超时：3 秒。
- 首 token 超时：15 秒。
- 总响应超时：60 秒。
- 管理端测试连接超时：10 秒。

重试规则：

- 只对连接失败、连接超时、HTTP 502、HTTP 503、HTTP 504 做有限重试。
- 最多重试 1 到 2 次。
- 流式响应已经开始输出后禁止重试。
- 非幂等操作禁止自动重试。

容错规则：

- Agent Runtime 不可用时返回明确错误码。
- 线程池满时返回 `AGENT_BUSY`。
- 调用超时时返回 `AGENT_TIMEOUT`。
- 权限不足返回 `AGENT_FORBIDDEN`。
- 额度不足返回 `QUOTA_NOT_ENOUGH`。

记录规则：

- 每次调用生成 `requestId`。
- 调用开始、成功、失败、超时都要记录。
- 调用失败也要记录调用记录。
- 扣减额度必须和调用记录状态一致。

# 十、额度与角色控制

## 10.1 核心规则

- 用户能否使用某 Agent，由角色策略和用户覆盖配置共同决定。
- 用户额度先看用户级覆盖配置。
- 没有用户级覆盖配置时，使用角色默认策略。
- 管理员可以按用户调整某 Agent 的额度。
- 后台必须能查询调用记录和额度消耗。

## 10.2 数据对象

`role_agent_policy`：

- `id`
- `role_id`
- `agent_id`
- `enabled`
- `daily_limit`
- `monthly_limit`
- `total_limit`

`user_agent_quota`：

- `id`
- `user_id`
- `agent_id`
- `daily_limit`
- `monthly_limit`
- `total_limit`
- `daily_used`
- `monthly_used`
- `total_used`
- `version`

`agent_call_record`：

- `id`
- `request_id`
- `user_id`
- `agent_id`
- `session_id`
- `status`
- `prompt_tokens`
- `completion_tokens`
- `total_tokens`
- `cost_quota`
- `error_code`
- `error_message`
- `started_at`
- `finished_at`

## 10.3 调用判断流程

1. 校验用户是否登录。
2. 查询用户角色。
3. 查询角色是否允许使用目标 Agent。
4. 查询用户是否有单独额度配置。
5. 计算当前可用额度。
6. 额度不足则拒绝调用。
7. 创建调用记录，状态为 `RUNNING`。
8. 调用 Python Agent Runtime。
9. 调用成功则记录 token 和消耗。
10. 扣减额度。
11. 更新调用记录为 `SUCCESS`。
12. 调用失败则更新调用记录为 `FAILED` 或 `TIMEOUT`。

## 10.4 代码组织

额度逻辑统一放在 `quota` 模块。

`agent` 模块只负责调用流程编排，不直接操作额度表。

必须提供 Service：

- `QuotaCheckService`
- `QuotaDeductService`
- `QuotaAdminService`
- `AgentCallRecordService`

# 十一、前端代码组织规范

## 11.1 总目录结构

前端推荐结构：

```text
src
├── app
├── modules
├── components
│   └── ui
├── lib
├── hooks
├── styles
└── types
```

职责：

- `app`：Next.js 路由、布局、页面入口。
- `modules`：业务模块。
- `components/ui`：shadcn/ui 基础组件。
- `components/layout`：布局组件。
- `lib`：请求封装、工具、常量。
- `hooks`：跨模块通用 hook。
- `types`：跨模块通用类型。

## 11.2 业务模块划分

```text
modules
├── auth
├── dashboard
├── content
├── agent
├── quota
├── user
├── file
└── system
```

模块职责：

- `auth`：登录、注册、当前用户。
- `dashboard`：后台布局、后台首页。
- `content`：文章、分类、标签。
- `agent`：Agent 列表、详情、体验、会话。
- `quota`：额度、调用记录。
- `user`：用户、角色、后台账号。
- `file`：文件上传和文件选择。
- `system`：菜单、字典、站点配置。

## 11.3 模块内部结构

每个模块内部统一结构：

```text
module
├── api
├── components
├── hooks
├── types
├── schemas
├── utils
└── constants
```

规则：

- `api` 只封装 HTTP 请求。
- `components` 只放模块内 UI 组件。
- `hooks` 只放模块内状态和数据逻辑。
- `types` 只放模块内类型。
- `schemas` 只放 zod 表单校验。
- `utils` 只放模块内纯函数。
- `constants` 只放模块内常量。

## 11.4 前端分层职责

页面组件：

- 负责路由入口。
- 负责组合模块组件。
- 禁止堆积复杂业务逻辑。

业务组件：

- 负责展示具体业务 UI。
- 可以调用模块 hooks。
- 禁止直接拼接后端 URL。

API 层：

- 只负责请求后端。
- 统一处理 `Result<T>`。
- 禁止包含 UI 逻辑。

Hook 层：

- 负责数据获取、表单状态、交互状态。
- 禁止写大型 JSX。

## 11.5 跨模块调用规则

允许：

```text
page -> module component
module component -> same module hook
module hook -> same module api
module api -> lib/request
module -> shared components/ui
```

禁止：

```text
module api -> another module api
component -> fetch directly
component -> database field assumption
frontend -> python-agent-runtime
frontend -> mysql / redis
```

## 11.6 Server Component 与 Client Component

默认使用 Server Component。

必须使用 Client Component 的场景：

- 表单输入。
- 弹窗。
- 下拉菜单。
- SSE 连接。
- 浏览器事件。
- 本地状态。
- `useEffect`、`useState`、`useRef`。

规则：

- `use client` 文件越小越好。
- 不要把整个页面都做成 Client Component，除非确实需要。
- Server Component 负责静态数据和页面骨架。
- Client Component 负责交互。

## 11.6.1 前端 UI 组件使用规范

- 所有页面中的基础交互控件，默认必须来自 `src/components/ui`。
- 适用范围至少包括：`Button`、`Input`、`Textarea`、`Select`、`Checkbox`、`Dialog`、`Sheet`、`Tabs`、`Table`、`Pagination`、`Badge`、`Tooltip`、`Skeleton`、`Sidebar`、`Toaster`。
- 业务组件可以组合这些原生组件，但不能重新发明一套同职责基础组件。
- 允许做主题色、间距、布局级样式调整，但不允许把官方组件行为替换成手写 DOM + 自定义交互。
- 如果只是为了改视觉，不允许引入第二套 UI 库。
- 表单字段校验提示允许保留在字段附近，但表单提交成功、接口失败、批量操作结果等全局反馈不放在页面里硬编码提示条，统一走全局提示体系。

## 11.6.2 前端提示信息规范

- 全局非阻塞提示统一使用 `sonner`，统一由根布局挂载全局 `Toaster`。
- 当前项目统一提示位置为顶部居中，即 `top-center`。
- 当前项目统一使用 `sonner` 的实心色块风格（solid / richColors），不使用浅色背景提示样式。
- V1 不显示 toast 关闭叉号，不提供右上角关闭按钮。
- 成功提示统一使用 `toast.success`。
- 错误提示统一使用 `toast.error`。
- 警告或提醒类提示统一使用 `toast.warning`。
- 如果只是普通说明而非成功、失败、警告，可使用默认 `toast(...)`，但必须谨慎，避免滥用。
- 后端返回错误码或错误消息时，前端必须先转换成用户可理解文案，再决定是否弹出 toast。
- 表单字段级校验错误优先显示在字段附近，不用 toast 替代字段校验。
- 删除确认、危险操作确认属于阻塞交互，必须使用框架原生确认类对话框，不允许用 toast 代替确认弹窗。

## 11.6.3 前端常见异常场景处理规范

异常处理必须遵守“三层分工”：

- 页面级问题由页面承接，例如 `404`、`403`、全局崩溃。
- 操作结果由 toast 承接，例如保存成功、删除失败、批量操作部分成功。
- 字段错误由表单承接，例如必填缺失、格式错误、长度错误。

常见场景处理规则：

1. 未登录访问受保护页面：
- 前端收到 `401` 后，必须清理本地登录态。
- 必须跳转到 `/login?redirect=当前页地址`。
- 可使用 `toast.warning("请先登录")` 或等价文案。

2. token 过期、token 无效、登录失效：
- 前端收到 `401` 后，必须清理本地 token 和用户缓存。
- 必须跳转到登录页，并保留 redirect。
- 提示文案统一为“登录已过期，请重新登录”这一类用户可理解文案。

3. 已登录但权限不足：
- 前端收到 `403` 后，不跳登录页。
- 必须进入站内无权限页面，或展示明确的无权限态。
- 页面应提供“返回首页”“返回上一页”“返回个人页/仪表盘”等入口。

4. 访问不存在的前端路由：
- 必须使用 App Router 的 `not-found.tsx` 承接。
- 不允许仅使用 toast 提示“页面不存在”。
- V1 最简实现为：
  - 前台单独一套 `404`
  - 后台单独一套 `404`
  - 后台单独一套 `403`
  - `401` 不做独立页面，直接跳登录页
- 文案与入口最简约定：
  - 前台 `404`：
    - 标题：页面不存在
    - 说明：你访问的内容不存在，或已下架
    - 按钮：返回首页 / 去博客列表 / 去 Agent 专栏
  - 后台 `404`：
    - 标题：页面不存在
    - 说明：当前后台页面不存在或已被移除
    - 按钮：返回仪表盘 / 返回上一页
  - 后台 `403`：
    - 标题：无权限访问
    - 说明：当前账号没有访问该功能的权限
    - 按钮：返回仪表盘 / 返回上一页
- 页面结构应统一，但前台与后台文案按场景区分，不强行共用完全相同的话术。

5. 页面路由存在，但详情数据不存在：
- 详情页应进入 `404` 页面或使用 `notFound()`。
- 列表页或弹窗内的单次操作，如果目标记录已不存在，可使用 `toast.error` 提示“记录不存在或已删除”。

6. 表单字段校验失败：
- 必须优先显示在字段附近。
- 不允许只弹 toast 而不告诉用户具体哪个字段错了。

7. 请求参数不合法：
- 表单类请求优先回填或展示字段错误。
- 非表单请求可使用 `toast.error("请求参数不正确")` 或等价文案。

8. 业务规则类错误：
- 例如用户名重复、菜单编码重复、额度不足、不能删除自己、不能停用当前登录账号。
- 必须展示明确业务文案，禁止直接把后端原始异常堆栈展示给用户。

9. 批量操作部分成功、部分失败：
- 必须用 `toast.warning` 明确说明成功数量和首个失败原因。
- 不允许把“部分成功”直接归类成普通成功。

10. 网络断开、浏览器离线、请求超时：
- 应展示可理解提示，例如“网络连接不可用”“请求超时，请稍后重试”。
- 页面级关键请求失败时，应保留错误态或重试入口，不允许直接白屏。

11. 服务端异常、网关异常、服务不可用：
- `500`、`502`、`503`、`504` 场景必须统一提示友好文案。
- 禁止直接向用户透出后端堆栈、SQL、内部类名或第三方原始错误。

12. Next.js 运行时异常和构建资源异常：
- 必须提供全局错误页，例如 `error.tsx` 或 `global-error.tsx`。
- `/_next/static/chunks/...` 这类 chunk 资源 404 可自动刷新恢复一次。
- 自动恢复失败后，必须进入明确错误页，不能无限刷新。

13. SSE / Agent 会话异常：
- 必须区分“连接失败”“响应中断”“调用超时”“服务繁忙”。
- 已输出的内容在中断时必须保留，不允许整段清空。
- 消息状态必须从“生成中”切到“已中断”或等价状态，不允许一直停留在加载中。
- 中断后必须给出明确原因文案，例如“响应中断，请重试”“响应超时，已停止生成”“额度不足，响应已停止”“服务繁忙，请稍后重试”。
- V1 最简恢复动作统一提供“重新生成”，不要求实现自动续流或“继续生成”。
- Agent 相关错误允许同时使用页面状态提示和 toast。

## 11.7 Agent 前端交互规范

- Agent 列表页只展示基础信息。
- Agent 详情页展示说明、适用场景、额度信息和体验入口。
- 用户未登录时点击体验必须跳转登录。
- SSE 消息必须支持加载中、输出中、完成、失败、超时状态。
- 客户端断开时必须关闭 EventSource 或 fetch stream。
- 错误信息必须使用后端错误码映射成用户可理解文案。
- SSE 中断后的最简实现：
  - 保留已输出的半段内容。
  - 将消息状态标记为 `INTERRUPTED`。
  - 持久化半段内容与中断状态，刷新页面后仍可见。
  - 提供“重新生成”入口，不要求实现自动续流。

## 11.8 后台管理页面规范

后台 CRUD 页面统一包含：

- 标题。
- 筛选区。
- 操作区。
- 表格。
- 分页。
- 新增弹窗或抽屉。
- 编辑弹窗或抽屉。
- 删除确认。

后台表格规则：

- 列字段必须来自接口 VO。
- 禁止直接使用数据库字段假设。
- 时间字段统一格式化。
- 状态字段使用 Badge。
- 删除操作必须二次确认。

# 十二、接口规范

## 12.1 通用规则

路径风格：

```text
/api/v1/{资源复数名}
```

CRUD 示例：

```text
GET    /api/v1/articles
POST   /api/v1/articles
GET    /api/v1/articles/{id}
PUT    /api/v1/articles/{id}
DELETE /api/v1/articles/{id}
```

非 CRUD 操作用动词：

```text
POST /api/v1/agents/{id}/test-connection
POST /api/v1/agents/{id}/chat
POST /api/v1/files/upload
```

统一响应：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

分页请求：

```text
page: 从 1 开始
pageSize: 默认 20，最大 100
```

分页响应：

```text
Result<PageResult<T>>
PageResult: list, total, page, pageSize
```

空值规则：

- 列表字段空时返回 `[]`。
- 字符串字段空时返回 `""`。
- 对象不存在时返回 `null`。
- 后端内部可以使用 `null` 表示缺失，但接口输出必须稳定。

错误码分段：

- `1000-1999`：通用错误。
- `2000-2999`：认证和用户。
- `3000-3999`：博客内容。
- `4000-4999`：Agent。
- `5000-5999`：额度。
- `6000-6999`：文件。
- `7000-7999`：系统配置。

## 12.1.1 HTTP 状态码与前端处理语义

后端除了返回统一 `Result<T>` 业务结构外，HTTP 状态语义也必须正确，不允许把所有异常都包成 HTTP `200`。

推荐约定：

- `200 OK`：请求成功，业务成功。
- `400 Bad Request`：请求参数不合法、参数格式错误。
- `401 Unauthorized`：未登录、token 缺失、token 无效、token 已过期。
- `403 Forbidden`：已登录，但没有权限访问目标资源或功能。
- `404 Not Found`：接口不存在，或请求的资源不存在。
- `409 Conflict`：资源冲突，例如用户名已存在、编码已存在、重复创建。
- `422 Unprocessable Entity`：业务校验失败但语法正确，可按实际情况选用。
- `429 Too Many Requests`：限流、请求过于频繁。
- `500 Internal Server Error`：服务端未预期异常。
- `502 Bad Gateway`：上游服务异常。
- `503 Service Unavailable`：服务暂时不可用、线程池繁忙、Agent Runtime 不可用。
- `504 Gateway Timeout`：上游调用超时、Agent 响应超时。

前端处理规则：

- `401`：清理登录态，跳登录页，保留 redirect。
- `403`：进入无权限页或无权限态，不跳登录页。
- `404`：页面类进入站内 404，接口类按详情页或操作型场景分别处理。
- `409`：优先提示冲突原因，必要时绑定到具体字段。
- `500/502/503/504`：统一展示友好错误提示，并提供刷新或重试能力。

## 12.1.2 V1 最小实现补充约定

1. 角色模型：
- V1 统一使用单角色模型。
- 用户只绑定一个 `roleCode`，不使用多角色 `roleIds` 模型。

2. Agent 调用记录状态：
- V1 最简状态统一使用：
  - `SUCCESS`
  - `FAILED`
  - `TIMEOUT`
  - `INTERRUPTED`
- 更细的原因统一通过 `errorCode`、`errorMessage` 承接，例如：
  - `QUOTA_NOT_ENOUGH`
  - `AGENT_BUSY`
  - `NETWORK_ERROR`
  - `PROVIDER_ERROR`
  - `USER_CANCELLED`
- V1 Agent 统一采用 provider 化后端代理调用：
  - `providerType = CUSTOM`：使用 `runtimeUrl`，由后端代理调用自定义 Agent Runtime。
  - `providerType = DIFY`：使用 `apiUrl`、`apiKey`、`responseMode`，由后端代理调用 Dify API。
  - V1 Dify 只支持 `responseMode = blocking`，暂不接入 Dify streaming。
  - 前端统一调用 `POST /api/v1/agents/{id}/chat`，不直接调用 Python Runtime、Dify API 或 iframe 地址。
  - Dify API Key 只允许保存在后端或数据库，禁止返回给前端。
  - iframe 嵌入方式不再作为 V1 受额度控制 Agent 的实现方式。

3. 文件上传落地方式：
- V1 文件上传统一使用本地磁盘。
- 存储目录推荐为 `storage/uploads/年/月/日/` 结构。
- 落盘文件名统一使用 `UUID + 原扩展名`，不直接使用原文件名。
- 对外访问 URL 统一使用静态暴露路径 `/uploads/年/月/日/UUID.ext`。
- 数据库至少记录原文件名、存储文件名、相对路径、访问 URL、文件大小、contentType。
- 删除文件时，V1 最简实现为删除数据库记录并同步删除物理文件；如果物理文件已不存在，不应导致整个删除接口失败。
- V1 不做文件引用检查，允许删除；前端被引用处统一兜底默认图。

4. 菜单驱动方式：
- V1 采用“前端静态路由 + 后端菜单树与权限控制”的最简实现。
- 页面组件和路由注册仍保留在前端。
- 后端负责下发菜单树、可见性和权限标识。
- 前端根据后端菜单树渲染侧边栏、tab、面包屑和按钮显隐。

5. 按钮级权限：
- V1 纳入按钮级权限，但只做最简显示控制。
- 前端根据权限集合控制按钮显示，无权限按钮直接隐藏。
- 后端关键接口必须保留服务端权限校验，不能只依赖前端隐藏按钮。

6. 用户注册最简规则：
- 用户名唯一。
- 昵称可重复。
- 邮箱可为空。
- 密码最小长度为 6 位。
- 前台注册默认 `roleCode = USER`。
- 前端有确认密码字段，后端只接收一个 `password`。

7. 前台可见性最简规则：
- 文章或 Agent 处于草稿状态时，前台不可见。
- 文章或 Agent 处于下架状态时，前台不可见。
- 前台访问草稿或下架内容时，统一按不存在处理，即 `404`。

8. 额度最简规则：
- V1 额度按次扣减，不按 token 扣减。
- 默认额度统一为总计 10 次，不区分每日或每月额度。
- 请求开始前必须先检查是否还有剩余额度。
- 只有剩余额度大于 0 时，才允许本次请求开始。
- 额度扣减时机统一为请求成功后扣减，请求失败不扣减。
- V1 不做流式过程中按次额度中断。

9. 登录态最简规则：
- V1 登录态统一采用最简长时效实现。
- token 默认有效期按 1 个月处理。
- 不引入 refresh token。
- 不引入滑动续期。

10. 文件上传最简限制：
- V1 文件上传只允许 `jpg`、`jpeg`、`png`、`webp`。
- 单文件大小限制统一为 5MB。

11. 用户名与昵称最简规则：
- 用户名只允许字母、数字、下划线。
- 用户名长度统一为 4 到 20 位。
- 昵称最大长度统一为 20。

12. 默认图最简规则：
- 文章封面、Agent 封面、头像等兜底默认图统一放在前端静态资源目录。
- 被删除文件或空封面场景统一回退到前端默认图。

13. Tabs 持久化最简规则：
- 后台 tabs 统一使用 `localStorage` 做持久化。
- 存储内容只包含 `tabs` 和 `activeTabKey`，不保存页面内部复杂状态。
- 存储 key 按项目和用户隔离。
- 登录后恢复，退出登录时清空。

14. 删除确认最简规则：
- 删除确认统一使用前端 UI 框架的 `AlertDialog`。
- 不再使用 `window.confirm`。
- 单条删除和批量删除统一走同一套确认交互。

15. 上级菜单选择最简规则：
- 上级菜单统一采用 `Popover + Command + 树形缩进列表` 实现。
- 菜单树先拍平成列表，再按层级做缩进展示。
- 表单实际提交值仍然是 `parentId`。
- 统一提供“顶级菜单”选项，对应 `parentId = 0`。
- 编辑菜单时必须禁选自己。
- 编辑菜单时必须禁选自己的全部子孙节点。

16. Remember Me 最简规则：
- V1 直接去掉 `Remember Me` 复选框。
- 不保留无实际差异的假交互。

17. 默认图组件最简规则：
- 头像统一使用 `AvatarFallback` 思路实现兜底。
- 普通图片统一封装业务层 `AppImage` 或等价组件处理 fallback。
- 不允许在各页面零散手写图片加载失败兜底逻辑。
- 默认图资源路径统一为：
  - `/images/defaults/article-cover.png`
  - `/images/defaults/agent-cover.png`
  - `/images/defaults/avatar.png`

18. 菜单树接口最简返回结构：
- 后端统一返回：
  - 路径：`GET /api/v1/me/navigation`
  - `menus: MenuTreeNode[]`
  - `permissions: string[]`
- `MenuTreeNode` 最小字段统一包含：
  - `id`
  - `parentId`
  - `name`
  - `title`
  - `path`
  - `component`
  - `icon`
  - `type`
  - `permission`
  - `visible`
  - `status`
  - `sortOrder`
  - `children`
- 后端应按当前用户权限预先过滤菜单树，前端不再做复杂权限裁剪。
- `icon` 字段统一传 Lucide 图标名字符串。
- 后端允许保存任意图标字符串。
- 前端统一通过白名单 `iconMap` 做映射，未命中时回退到默认图标。
- `component` 字段保留在菜单数据中，但前端不直接用它动态渲染页面，只作为菜单元数据和管理展示用途。
- 后端校验最简规则：
  - `CATALOG`：`component` 可空
  - `MENU`：`component` 必填
  - `BUTTON`：`component` 为空

19. 图片上传返回值最简规则：
- 上传接口统一返回：
  - `id`
  - `url`
  - `filename`
  - `size`
  - `contentType`
- V1 业务表单字段直接保存 `url`。
- 文件管理模块可以单独维护 `id` 与文件记录。

20. 前台页面开发拆分最简规则：
- 前台博客与 Agent 页面按用户访问路径拆任务，不按技术层过度拆分。
- 推荐顺序：
  - 第一阶段：公共层
    - 前台布局
    - 公共卡片
    - 默认图组件
    - 分页 / 空态 / 404 组件
  - 第二阶段：博客线
    - `/blog`
    - `/blog/posts`
    - `/blog/posts/[id]`
  - 第三阶段：Agent 线
    - `/blog/agent`
    - `/blog/agent/[id]`
  - 第四阶段：体验链路
    - Agent 登录拦截
    - 会话区
    - SSE 中断 / 超时 / 失败 / 重试

# 十三、性能瓶颈预判

按严重程度排序：

1. Agent 长连接占用线程。
触发条件：多个用户同时调用慢 Agent，Python Runtime 或 LLM 30 秒以上不返回。
一期处理：必须处理。使用独立线程池、超时、SSE 断开清理。

2. Python Agent Runtime 不稳定。
触发条件：某个 Agent 崩溃、卡死、依赖外部服务超时。
一期处理：必须处理。Spring Boot 侧设置超时、熔断、错误码和调用记录。

3. MySQL 调用记录表增长。
触发条件：Agent 调用量增加，后台无时间范围查询。
一期处理：部分处理。加时间索引，后台强制时间筛选，暂不分表。

4. 前端后台页面包体过大。
触发条件：混用多个 UI 框架、引入大型组件、首屏加载 dashboard。
一期处理：必须处理。统一 shadcn/Radix/Tailwind/lucide，移除重复组件体系。

5. 文件上传占用应用资源。
触发条件：上传大图或多文件。
一期处理：简单处理。限制文件大小和类型，后期接对象存储。

# 十四、拆分演进预期

当前是 Spring Boot 单体应用。

如果未来拆成独立服务，按本文档规范改动量可控。

优先可拆模块：

- Agent Runtime：已经独立。
- 文件服务：可从 `file` 模块拆出。
- 内容服务：文章、分类、标签可独立。
- 额度与计量服务：当 Agent 调用量上升时可独立。

当前必须遵守的拆分前提：

- 模块之间只能通过 Service 调用。
- Controller 不跨模块调 Service。
- Service 不跨模块调 Mapper。
- 数据对象不跨层泄露。
- 外部调用都放 `infrastructure`。

预估改动量：

- 如果严格遵守本文档，拆分主要是 Controller、Service 接口、数据源和远程调用改造。
- 如果随意跨模块调 Mapper 或共享 Entity，拆分成本会显著上升。

# 十六、行为指令

## 16.1 写代码时

- 每个功能用最简单直接的方式实现。
- 不引入不必要的设计模式，除非用户明确要求。
- 不做过度抽象。
- 不引入技术栈以外的依赖，需要时先问用户。
- 所有外部调用必须有超时设置。
- 所有外部调用必须有日志和错误码。
- 所有配置项必须外化到 `application.yml` 或环境变量。
- 禁止硬编码密码、Token、密钥、外部服务地址。
- Controller 不写业务规则。
- Service 不返回 Entity 给 Controller。
- Mapper 不写业务规则。
- 前端组件不直接请求 Python Runtime。
- 前端组件不直接拼接后端 URL。

## 16.2 改代码时

- 先阅读本文档和相关模块代码。
- 先理解相关模块的设计意图。
- 不要为了新功能破坏已有接口契约。
- 不要随意改接口字段名。
- 不要随意改数据库字段名。
- 不要破坏已有页面路径。
- 改完必须确认已有测试通过。
- 如果没有测试，至少说明未运行测试的原因。
- 默认按最低验收标准执行：
  - 前端至少执行 `pnpm build`
  - 后端至少执行 `mvn -DskipTests package`
- 以下高风险改动必须额外做手工 smoke test：
  - 登录 / 退出登录
  - 权限 / 菜单 / 路由
  - 文件上传
  - Agent / SSE
  - 额度扣减

## 16.3 不确定时

- 架构选择给出 2 到 3 个方案对比，由用户拍板。
- 规范没覆盖的情况，先问用户，不要自行编规矩。
- 遇到已有实现和本文档冲突时，优先按本文档规划新代码，并说明冲突点。
- 遇到性能、稳定性、安全性三者冲突时，优先保证安全性和稳定性。

## 16.4 禁止事项

- 禁止为了快速实现直接绕过权限和额度。
- 禁止前端保存后端密钥。
- 禁止 Python Runtime 直接暴露公网。
- 禁止无超时调用外部服务。
- 禁止无日志吞异常。
- 禁止无条件更新和删除数据库数据。
- 禁止引入第二套大型 UI 框架。
- 禁止在没有用户确认的情况下改动核心技术栈。
